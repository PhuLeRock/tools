Helm 3 khác với Helm 2 chỗ nào ?

/DevOps /Helm 3 khác với Helm 2 chỗ nào ?
  • June 6, 2019
  • phuletv
  • DevOps

Written by devops Trần Nguyên Bảo.

Tóm lại Helm 3.0.0-alpha.1 có những điểm này nổi bật:

 

1. Bỏ tiller
Lý do:
+ K8S v1.6 enables RBAC by default, gây khó khăn cho việc quản lý permission của helm trên cluster vì tiller cần có 1 service account để interact với K8S API Server
+ Cách quản lý release của Tiller ko còn phù hợp nữa vì đang quản lý ở phạm vi toàn cluster, tức là 1 release đã thực hiện trên 1 namespace thì ko thể deploy release đó trên namespace khác

2. Release name được quản lý theo scope của Namespace
Vì lý do 2 ở trên nên v3 này sẽ có thể deploy cùng 1 release trên nhiều namespace khác nhau, quá xá hay!

3. Hợp nhất requirements.yaml thành Chart.yaml
requirements.yaml là dùng để định nghĩa những dependencies của 1 release
requirements.lock là dùng để khóa những thay dependencies của 1 release khi command này (helm dep build) được chạy, cái này anh em nào làm việc với packages sẽ hiểu, khóa lại là để những dependencies consistency trên nhiều môi trường khác nhau, quản lý những version 1 cách strictly

helm 3 khi chạy “helm dep build” sẽ ko đọc cái requirements.lock nữa, thay vào đó helm 3 sẽ gộp requirements.yaml vào Chart.yaml

4. Bắt phải có tham số –generate-name
Helm 2 khi run helm install sẽ generate ra 1 name (auto-generated name)
Helm 3 nếu vẫn muốn có auto-generated name thì thêm –generate-name

5. Bỏ helm serve command

6. Hỗ trợ library chart
Library chart là chart được chia sẽ cho những chart khác mà không tạo ra bất kỳ thông tin release nào trong helm
Định nghĩa ở trong Chart.yaml như requirements.yaml nói ở trên

7. Thay đổi tên 1 vài commands
+ helm delete -> helm uninstall (nhưng vẫn có thể dùng 1 trong 2 – backward compatibility)
+ Helm 2 để xóa sạch release (bao gồm history) thì xài tham số –purge, Helm 3 enabled by default
Để ko xóa sạch release mà ko xóa history thì dùng command helm uninstall –keep-history
+ helm inspect -> helm show
+ helm fetch -> helm pull
